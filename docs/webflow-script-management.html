Webflow Script Management

The core idea

Every tracking script (Google Analytics, Meta Pixel, HotJar, etc.) needs visitor consent before it runs. To enforce that, we use two tools together:

GTM (Google Tag Manager) is a container that holds all tracking scripts. Instead of pasting scripts directly into Webflow, we put them inside GTM as "tags." GTM controls when each tag fires - and it checks consent before firing.

Cookiebot shows the cookie consent banner. It lives inside the GTM container as a special tag. When a visitor accepts or declines, Cookiebot tells GTM which categories are allowed (statistics, marketing, preferences), and GTM only fires the matching tags.

The only script in the Webflow site header should be the GTM snippet. Cookiebot loads through GTM, and all tracking loads through GTM. One script controls everything.

The flow: visitor loads page, GTM loads, Cookiebot shows the banner, visitor chooses, GTM fires only the allowed tags.

Cookiebot fires first not because of its position in a list, but because GTM has built-in trigger priorities. The "Consent Initialization" trigger is guaranteed to run before "Initialization", "All Pages", and all other triggers. That's what makes the consent flow work.

Tracking scripts vs other scripts

Not everything in Webflow's custom code boxes is a tracking script. They're handled differently:

Tracking scripts collect data about visitors - analytics, advertising pixels, chat widgets that log behavior, heatmaps. These need consent and must live inside GTM. Examples: Google Analytics (gtag.js), Meta Pixel (fbevents.js), HotJar, LinkedIn Insight Tag, Google Ads conversion tracking.

Non-tracking scripts don't collect visitor data. These do NOT go into GTM - they stay in Webflow, ideally managed via the API. Examples: CSS fixes (like the font smoothing snippet on Väst Förvaltning), custom fonts, JSON-LD structured data, accessibility tools, Webflow's own framework scripts.

What the audit app can do (and what needs to be done by hand)

Cookiebot

Cookiebot is mostly self-managing. Once it's set up for a client (at cookiebot.com), it scans the site monthly, categorizes all cookies it finds, and shows the consent banner automatically. We never need to write anything to Cookiebot.

The audit app can read everything about a client's Cookiebot setup without any login or API key. It just needs the Cookiebot ID (a long code like 1a2b3c4d-5e6f-...). With that ID, the app fetches all cookie data from a public Cookiebot endpoint - every cookie the site sets, which category it's in (necessary, statistics, marketing, preferences), its expiry, and its provider. This is what powers the C1-C5 checks in the audit.

The app can also detect the Cookiebot ID automatically when you scan a site - it looks for the Cookiebot script tag in the site's HTML. If Cookiebot is loaded through GTM instead of directly in the HTML (which is the recommended setup), the app can find the ID inside the GTM container, but that requires the GTM API connection (see below).

GTM (Google Tag Manager)

The audit app can detect which GTM container a site uses (e.g. GTM-ABC1234) just by scanning the site's HTML - no API needed. This happens automatically when you click "Detect IDs" or run a scan.

But knowing the container ID is different from seeing what's inside it. To check whether Cookiebot is set up correctly inside GTM, whether tracking tags have consent triggers, or which tags exist at all - the app needs to connect to the GTM API. This requires an OAuth token (the one we're waiting on your boss's 2FA code for). Once connected, the app reads the container contents and runs those checks automatically during a scan.

Creating or editing tags inside GTM is done manually at tagmanager.google.com. The GTM API technically supports writing, but changing a client's tag configuration programmatically is too risky - a mistake could break their analytics or consent flow. Same for creating new GTM containers.

Webflow scripts

Webflow has two ways to add scripts, and the difference matters a lot:

Both manually-pasted and API-registered scripts end up in the same place - the custom code boxes in the Designer. They look the same on the live site too. The difference is on our side: the API can only read and manage scripts that were added through the API. Scripts that someone pasted in manually are invisible to the API, even though they're sitting right next to API-added scripts in the same box.

That's why we want all scripts added through the API going forward. They're all in the same custom code box either way, but the API-added ones we can manage from the audit app.

How a correctly set up site looks

Whether it's a new site or an existing one after cleanup, the end state is the same:

  1. Custom code boxes in the Designer: contain only the GTM snippet (and optionally JSON-LD structured data). No tracking scripts, no loose analytics or pixel code. The GTM snippet was added via the API so we can manage it from the audit app - but it shows up in the custom code box just like any manually-pasted script would.
  2. GTM container: Cookiebot CMP tag (on Consent Initialization trigger) + all tracking tags with consent triggers.
  3. Cookiebot: manages itself - scans the site monthly, categorizes cookies, shows the banner.

New sites

For a brand new client site, the setup is straightforward because there's nothing to clean up:

  1. Make sure the client has a GTM container (at tagmanager.google.com) and a Cookiebot subscription (at cookiebot.com). If not, that needs to happen first.
  2. Inside the GTM container, set up the Cookiebot CMP tag on the "Consent Initialization - All Pages" trigger, with Consent Mode V2 enabled.
  3. Add any tracking tags (GA4, etc.) to the GTM container with proper consent triggers.
  4. Push the GTM snippet to the Webflow site header via the audit app / MCP.
  5. Never paste anything into the Designer custom code boxes. All tracking goes into GTM, all other scripts go via the API.
  6. Scan in the audit app to verify everything is detected and compliant.

Existing sites (cleanup)

Most of our sites already have scripts pasted manually into the Designer custom code boxes. This is the messy part, because we can't touch those scripts through code - it's all manual work in the Webflow Designer.

The order matters. You can't remove tracking scripts from Webflow until they're set up in GTM, otherwise you'd delete the client's analytics with nothing to replace it.

  1. Scan the site in the audit app. This identifies every script in the head and footer, flags which ones are tracking scripts, and detects the GTM/Cookiebot IDs.
  2. Sort out what's what. Look at the scan results. For each script found, decide: is this a tracking script (needs to move into GTM) or a non-tracking script (stays in Webflow, ideally via API later)?
  3. Recreate each tracking script as a GTM tag. Open tagmanager.google.com, go to the client's container, and for each tracking script found in Webflow, create a new tag with the right consent trigger. For example: if there's a loose GA4 script in the head, create a Google Tag in GTM with the same measurement ID.
  4. Verify in GTM Preview mode that the new tags fire correctly. Don't skip this - if a tag is misconfigured, you'll lose the client's data after the next step.
  5. Comment out tracking scripts in Webflow Designer. Open Site Settings > Custom Code. Don't delete the tracking scripts yet - wrap them in <!-- --> HTML comments instead. This way they stop running but you can restore them quickly if something goes wrong. Leave non-tracking scripts (CSS, fonts, JSON-LD) as they are.
  6. Comment out the GTM snippet too if there's a manually-added one in the Designer custom code. We'll push an API-managed one in the next step, and having two would cause double-tracking.
  7. Push the GTM snippet via the audit app / MCP. This adds a single, API-managed GTM snippet to the site header.
  8. Re-scan and verify everything works - no loose tracking scripts firing, GTM snippet present, Cookiebot detected, all tracking coming through GTM.
  9. After a few weeks, once you've confirmed everything works correctly, go back and delete the commented-out scripts from the Designer custom code boxes.

Important: Don't remove scripts from Webflow until they exist in GTM. If you delete a tracking script before recreating it as a GTM tag, the client loses that tracking data with no way to recover it.
