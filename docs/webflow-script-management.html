Webflow Script Management

How scripts, consent, and the audit app work together

The core idea

Every tracking script (Google Analytics, Meta Pixel, HotJar, etc.) needs visitor consent before it runs. To enforce that, we use two tools together:

GTM (Google Tag Manager) A container that holds all tracking scripts. Instead of pasting scripts directly into Webflow, we put them inside GTM as "tags." GTM controls when each tag fires - and it checks consent before firing.
Cookiebot Shows the cookie consent banner. Lives inside the GTM container as a special tag. When a visitor accepts or declines, Cookiebot tells GTM which categories are allowed (statistics, marketing, preferences), and GTM only fires the matching tags.

The only script in the Webflow site header should be the GTM snippet. Cookiebot loads through GTM, and all tracking loads through GTM. One script controls everything.

Visitor loads page → GTM loads → Cookiebot shows banner → Visitor chooses → GTM fires allowed tags

Cookiebot fires first not because of its position in a list, but because GTM has built-in trigger priorities. The "Consent Initialization" trigger is guaranteed to run before "Initialization", "All Pages", and all other triggers. That's what makes the consent flow work.

What the audit app can do (and what needs to be done by hand)

Cookiebot

Cookiebot is mostly self-managing. Once it's set up for a client (at cookiebot.com), it scans the site monthly, categorizes all cookies it finds, and shows the consent banner automatically. We never need to write anything to Cookiebot.

The audit app can read everything about a client's Cookiebot setup without any login or API key. It just needs the Cookiebot ID (a long code like 1a2b3c4d-5e6f-...). With that ID, the app fetches all cookie data from a public Cookiebot endpoint - every cookie the site sets, which category it's in (necessary, statistics, marketing, preferences), its expiry, and its provider. This is what powers the C1-C5 checks in the audit.

The app can also detect the Cookiebot ID automatically when you scan a site - it looks for the Cookiebot script tag in the site's HTML. If Cookiebot is loaded through GTM instead of directly in the HTML (which is the recommended setup), the app can find the ID inside the GTM container, but that requires the GTM API connection (see below).

GTM (Google Tag Manager)

The audit app can detect which GTM container a site uses (e.g. GTM-ABC1234) just by scanning the site's HTML - no API needed. This happens automatically when you click "Detect IDs" or run a scan.

But knowing the container ID is different from seeing what's inside it. To check whether Cookiebot is set up correctly inside GTM, whether tracking tags have consent triggers, or which tags exist at all - the app needs to connect to the GTM API. This requires an OAuth token (the one we're waiting on your boss's 2FA code for). Once connected, the app reads the container contents and runs those checks automatically during a scan.

Creating or editing tags inside GTM is done manually at tagmanager.google.com. The GTM API technically supports writing, but changing a client's tag configuration programmatically is too risky - a mistake could break their analytics or consent flow. Same for creating new GTM containers.

Webflow scripts

Webflow has two ways to add scripts, and the difference matters a lot:

Designer custom code (the text boxes in Site Settings > Custom Code) is completely invisible to the API. If someone pasted a GTM snippet or a tracking script there manually, we cannot see it, edit it, or remove it through code. The only way to clean those up is to open the Webflow Designer and delete them by hand.

API-registered scripts (added through the Webflow Data API or our MCP connection) produce identical HTML on the live site, but we can read, update, and remove them from the audit app. This is why we want all scripts managed through the API after cleanup.

The only script the audit app pushes to Webflow is the GTM snippet - a standard script tag with the client's container ID. We don't create GTM containers or generate Cookiebot configs. Both of those already exist for the client (or need to be set up by hand at tagmanager.google.com and cookiebot.com). If a client has neither, the audit flags it as an issue.

Tracking scripts vs other scripts

Not everything in Webflow's custom code boxes is a tracking script. Before touching anything, you need to know which is which, because they're handled differently:

Tracking scripts collect data about visitors - analytics, advertising pixels, chat widgets that log behavior, heatmaps. These are the ones that need consent and must live inside GTM. Examples: Google Analytics (gtag.js), Meta Pixel (fbevents.js), HotJar, LinkedIn Insight Tag, Google Ads conversion tracking.

Non-tracking scripts don't collect visitor data. These do NOT go into GTM - they stay in Webflow, ideally managed via the API. Examples: CSS fixes (like the font smoothing snippet on Väst Förvaltning), custom fonts, JSON-LD structured data, accessibility tools, Webflow's own framework scripts.

The scan flags tracking scripts as issues. Non-tracking scripts are left alone (or noted for later API migration).

Cleaning up an existing site

The order matters here. You can't remove tracking scripts from Webflow until they're set up in GTM - otherwise you'd delete the client's analytics with nothing to replace it.

  1. Scan the site in the audit app. This identifies every script in the head and footer, flags which ones are tracking scripts, and detects the GTM/Cookiebot IDs. auto
  2. Sort out what's what. Look at the scan results. For each script found, decide: is this a tracking script (goes into GTM) or a non-tracking script (stays in Webflow, ideally via API)?
  3. Recreate each tracking script as a GTM tag. Open tagmanager.google.com, go to the client's container, and for each tracking script found in Webflow, create a new tag with the right consent trigger. For example: if there's a loose GA4 script in the head, create a Google Tag in GTM with the same measurement ID. manual in GTM
  4. Verify in GTM Preview mode that the new tags fire correctly. Don't skip this - if a tag is misconfigured, you'll lose the client's data after the next step.
  5. Remove tracking scripts from Webflow Designer. Open Site Settings > Custom Code, and remove only the tracking scripts you've already moved to GTM. Leave non-tracking scripts (CSS, fonts, JSON-LD) alone. Also check per-page custom code settings. manual in Designer
  6. Handle the GTM snippet itself. If there's already a manually-added GTM snippet in the Designer custom code, remove it too - otherwise you'll end up with two GTM snippets when we push the API-managed one. manual in Designer
  7. Push the GTM snippet via the audit app / MCP. This adds a single, API-managed GTM snippet to the site header. Since we removed the manual one in the previous step, there's exactly one. auto
  8. Re-scan to verify everything is clean - no loose tracking scripts, GTM snippet present, Cookiebot detected. auto
Never paste tracking scripts into Webflow's custom code boxes. They bypass consent and we can't manage them from the audit app. All tracking goes into GTM.
Don't remove scripts from Webflow until they exist in GTM. If you delete a tracking script before recreating it as a GTM tag, the client loses that tracking data with no way to recover it.

End state for each site

After cleanup, a properly configured site has:

  1. Designer custom code boxes: empty (nothing in head or footer)
  2. API-registered scripts: GTM snippet in header (+ JSON-LD if applicable)
  3. GTM container: Cookiebot CMP tag (on Consent Initialization trigger) + all tracking tags with consent triggers
  4. Cookiebot: manages itself - scans the site monthly, categorizes cookies, shows the banner
