Webflow Script Management

Process guide for the GDPR audit team - how to handle scripts on Webflow client sites

Background: Two Ways Scripts End Up on a Webflow Site

Designer custom code boxes API/MCP registered scripts
How added Someone pastes code into the Webflow Designer's head/body custom code fields A developer or tool registers and applies a script via the Webflow Data API or MCP server
Read via API No Yes - list_registered_scripts, list_applied_scripts
Edit/delete via API No Yes - delete_all_site_scripts, re-register with new version
Looks different in HTML? No - both produce identical <script> tags. You cannot tell them apart from the outside.
Key takeaway: All scripts on our existing client sites were pasted manually in the Designer. The API has zero access to those. To gain programmatic control, we need to remove the manual scripts and re-add them through the API/MCP.

What Goes Where

Script type Where it belongs Examples
Tracking / analytics / pixels Inside the GTM container as tags with consent triggers Google Analytics, Meta Pixel, LinkedIn Insight, HotJar, Microsoft Clarity, TikTok Pixel, HubSpot tracking
Consent management Webflow head (via API/MCP) - must load before GTM Cookiebot consent script
Tag management Webflow head (via API/MCP) - loads after Cookiebot GTM container snippet
Structured data Webflow head (via API/MCP or Designer) JSON-LD schema markup
Framework / platform scripts Left alone - managed by Webflow automatically webflow.js, webfont loader
CSS overrides for consent banner Webflow head (via API/MCP or Designer) Cookiebot banner styling
Rule: Never paste tracking scripts into Webflow's custom code boxes. They always go into GTM as tags with proper consent triggers. The only scripts in Webflow's head should be the Cookiebot snippet, the GTM snippet, JSON-LD, and framework scripts.

Flow: One-Time Cleanup (Existing Sites)

Every existing Webflow client site needs this process once to move from manual script management to the auditable, programmatic setup.

Step 1
Scan the site Automated
Run a scan in the GDPR audit app. It detects the GTM container ID and Cookiebot ID, flags any tracking scripts hardcoded outside GTM (checks A1, A2, B1, D1, D3), and identifies what needs to move.
Step 2
Verify/set up GTM container Automated + Manual review
Check the GTM container has: Cookiebot tag with correct cbid, consent triggers for each cookie category, Consent Mode V2 defaults. If the site has no GTM container yet, create one.
Step 3
Move tracking scripts into GTM Automated + Manual review
For each tracking script the scan flagged (Meta Pixel, LinkedIn, etc.), create a corresponding tag in the GTM container with the correct consent trigger. Verify the tag fires correctly.
Step 4
Clean up Webflow Designer Manual
Open the site in the Webflow Designer. Remove all tracking scripts from the head and body custom code boxes. Leave only: the GTM snippet, the Cookiebot snippet, JSON-LD structured data, and any Cookiebot CSS styling. This is the only step that cannot be automated.
Step 5
Migrate remaining scripts to API-registered Automated
Use the Webflow MCP (add_inline_site_script) to register the GTM snippet, Cookiebot snippet, and any other non-tracking head scripts as API-managed scripts. Then remove them from the Designer custom code boxes. From this point on, these scripts are fully manageable via the API.
Step 6
Re-scan and verify Automated
Run another scan to confirm: no tracking scripts outside GTM, Cookiebot consent banner loads, GTM container loads, Consent Mode V2 is active. All checks should pass.

Flow: New Client Sites

When onboarding a new Webflow site, start clean from day one.

Step 1
Set up GTM container
Create a GTM container for the client (or use an existing one). Add the Cookiebot tag with consent triggers and Consent Mode V2 defaults.
Step 2
Push scripts via Webflow MCP Automated
Use add_inline_site_script to register and apply the Cookiebot snippet and GTM snippet to the site. These are now API-managed from the start - no need to touch the Designer's custom code boxes.
Step 3
Add tracking via GTM only
All analytics, pixels, and tracking scripts go into the GTM container as tags with consent triggers. Nothing is ever pasted into Webflow's custom code.
Step 4
Scan and verify Automated
Run the GDPR audit scan to confirm compliance from the start.

For Colleagues: How to Add Scripts Going Forward

Do not paste scripts into Webflow's Designer custom code boxes. Scripts added there cannot be read, updated, or deleted through our tools. We lose all visibility and control.

If it's a tracking/analytics script:

  1. Log in to Google Tag Manager
  2. Open the client's GTM container
  3. Create a new tag for the script
  4. Set the appropriate consent trigger (statistics, marketing, etc.)
  5. Test in GTM's preview mode, then publish

If it's a non-tracking script (JSON-LD, CSS, structural):

  1. Ask a developer to add it via the Webflow API or MCP server using add_inline_site_script
  2. This registers the script so we can manage, update, and audit it programmatically
  3. Specify whether it goes in the header or footer

If you're unsure:

Ask before adding. The GDPR audit app will flag anything that's not compliant, but it's easier to do it right the first time.

Tools We Use

Tool What it does for us
GDPR Audit App (scanner) Scans live sites, detects GTM/Cookiebot IDs, flags non-compliant scripts, generates reports
Webflow MCP Server Lists all Webflow sites, registers/manages scripts via API, reads site metadata
GTM API (via OAuth2) Inspects GTM containers, lists tags, creates tags with consent triggers, looks up Cookiebot IDs
Cookiebot API Fetches cookie categorization data for a site using its Cookiebot ID
